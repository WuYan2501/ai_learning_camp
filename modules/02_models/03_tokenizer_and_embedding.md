# 2.3 Tokenizer 与 Embedding 🔥

> 很多人第一次学大模型，会直接跳到 Transformer，但其实在它之前还有两步非常关键：**文本怎么切、切完之后怎么变成模型能处理的向量。**
>
> 这一节就是把这条链路补完整：`文本 → token → 向量`。
>
> 返回 [模型层索引](/modules/02_models) | [知识库索引](/SYLLABUS)

---

## Tokenizer：文本怎么切成 token

- **需要懂的内容**：模型并不是直接“看懂汉字和单词”，而是先把文本切成一个个 token
- **简单点理解**：Tokenizer 像一个切分规则系统，负责把原始文本翻译成模型内部更适合处理的最小单元
- **为什么重要**：切分方式会直接影响序列长度、成本、中文效率、代码能力，甚至影响模型的使用体验

### 主流先知道这几类

- **BPE** ⭐
  - 从字符开始，不断把高频片段合并起来
  - 很多生成式模型都会用到这类思路
- **WordPiece**
  - 和 BPE 类似，但更偏向语言模型目标
  - 早期 BERT 体系常见
- **SentencePiece / Unigram**
  - 不依赖空格分词，适合中日韩等语言
  - 现在很多开源模型会用这类方案

### 你真正该关心什么

- **词表太小**：token 切得太碎，序列更长，算得更慢
- **词表太大**：Embedding 层会更大，训练和存储成本会上去
- **中文和代码特别敏感**：如果切分效率差，中文一句话 token 数偏多，使用成本会明显上升

---

## Embedding：token 怎么变成向量

- **需要懂的内容**：模型不能直接处理 token 编号，必须把它们映射成连续向量
- **简单点理解**：Embedding 就是在给每个 token 分配一个可学习的向量表示，让“意思相近”的内容在向量空间里更接近一些
- **为什么重要**：后面 Transformer 处理的不是单词本身，而是这些向量表示

---

## 静态表示 vs 上下文表示

- **静态词向量**
  - 同一个词不管放在哪，向量都一样
  - 代表路线：`Word2Vec`
- **上下文向量** ⭐
  - 同一个词放在不同句子里，向量可以不同
  - 这才是现代大模型真正强大的基础之一

- **为什么重要**：这意味着模型不是只会查词典，而是真的会根据上下文调整理解

---

## Sentence Embedding

- **需要懂的内容**：有时候我们关心的不是单个 token，而是整句话、整段文本的表示
- **典型用途**：RAG 检索、语义搜索、相似度匹配、聚类
- **和生成模型的关系**：生成模型擅长“写”，Embedding 模型更擅长“找”和“比”
- **简单点理解**：如果生成模型像会说话的人，那 Embedding 更像是很会做语义检索和匹配的“向量化理解器”

---

## 这部分别学偏了

- **不是所有向量都叫 Embedding 模型能力**：有些只是模型中间层表示，有些是专门为检索训练出来的句向量
- **不要一上来就陷进过多分词算法细节**：对当前学习主线来说，更重要的是知道切分方式会影响长度、成本、语言效率和下游效果
- **和应用层的关系**：后面学 RAG 时，你会再次遇到 Embedding，那时候重点会从“原理”转到“怎么选和怎么评估”
