## Agent 本质定义 ⭐

- **什么是 AI Agent**
  - *核心思想*：能够感知环境、自主决策、执行行动、并根据反馈迭代的 AI 系统
  - *与普通 LLM 的区别*：LLM 是「一问一答」，Agent 是「理解目标 → 拆解任务 → 自主执行 → 反思调整」
  - *核心能力*：自主性（Autonomy）+ 持续性（Persistence）+ 适应性（Adaptability）

- **Agent = LLM + 规划 + 记忆 + 工具 + 反思**
  ```
  感知(Perception) → 规划(Planning) → 记忆(Memory) → 工具调用(Tool Use) → 执行(Action) → 反思(Reflection)
  ```
  - *核心问题*：这六个模块中，哪个是 Agent 效果的最大瓶颈？（规划能力，因为它直接决定了 Agent 能否正确分解和执行复杂任务）

- **Agent 的三个层次**
  - **L1 - 工具增强型**：LLM + Function Calling，人驱动，AI 执行（如 ChatGPT Plugins）
  - **L2 - 半自主型**：LLM 自主规划和执行，人监督审批（如 Cursor、GitHub Copilot Workspace）
  - **L3 - 全自主型**：LLM 完全自主完成端到端任务（如 Devin、OpenAI Deep Research）
  - *核心问题*：当前大多数生产环境的 Agent 处于哪个层次？（L2，因为 LLM 的可靠性还不够支撑 L3）

## 规划范式 🔥🔥

### 反应式规划

- **ReAct** ⭐
  - *核心思想*：Thought → Action → Observation 循环，简单有效，大多数场景首选
  - *优势*：实时响应，每步都有推理依据，易于调试
  - *劣势*：短视（greedy），不做全局规划，长任务容易偏离目标
  - *适用场景*：步骤较少的任务（< 10步）、信息检索类任务

- **Reflexion** 🔥 ⭐
  - *核心思想*：Agent 完成任务后进行自我反思，将反思结果存入记忆，下次任务改进
  - *为什么重要*：让 Agent 有"从错误中学习"的能力
  - *实现*：执行 → 评估结果 → 生成反思文本 → 存入长期记忆 → 下次任务检索反思
  - *核心问题*：Reflexion 的反思质量如何保证？（依赖 LLM 的自我评估能力）

### 前瞻式规划

- **Plan-and-Execute** ⭐
  - *核心思想*：先生成完整计划，再逐步执行，执行时可动态修正
  - *vs ReAct*：ReAct 是边想边做，Plan-and-Execute 是先想后做
  - *适用场景*：长任务（> 10步）、需要全局视角的任务

- **Hierarchical Planning（层次化规划）** 🔥
  - *核心思想*：高层 Agent 分解为子目标，子 Agent 负责具体执行
  - *类比*：公司组织架构，CEO 定战略，VP 定策略，工程师执行
  - *实现*：LangGraph 的嵌套子图 / CrewAI 的分层任务

- **Dynamic Replanning（动态重规划）** 🔥 ⭐
  - *核心思想*：执行过程中发现计划不可行，动态调整计划
  - *触发条件*：工具调用失败 / 返回异常结果 / 超时 / 用户中断
  - *核心问题*：什么时候该重规划，什么时候该重试？（可恢复错误重试，不可恢复错误重规划）

### 搜索式规划

- **LATS（语言 Agent 树搜索）** 🔥
  - *核心思想*：将 MCTS（蒙特卡洛树搜索）引入 Agent，支持回溯和多路径探索
  - *优势*：能找到全局最优路径，而非贪心路径
  - *劣势*：成本高（每个节点都要 LLM 推理）
  - *适用场景*：高价值决策、需要最优解的场景

- **MCTS + LLM 推理** ⭐
  - *核心思想*：用 LLM 作为 MCTS 的评估函数和扩展策略
  - *应用*：代码生成（AlphaCode）/ 数学推理 / 游戏 AI
  - *核心问题*：LLM + MCTS 的搜索效率如何？（远优于暴力搜索，但仍然昂贵）

## 工具调用（Function Calling）🔥🔥 ⭐

### 基础原理

- *核心思想*：模型输出结构化的工具调用请求（JSON），外部执行后将结果返回给模型
- *实现原理*：SFT 阶段训练模型识别何时调用工具、如何格式化参数
- *核心问题*：模型怎么学会调工具？（训练数据中包含工具调用的示例，格式化为特殊 token）

### 进阶能力

- **并行工具调用**：一次输出多个工具调用，并行执行，减少延迟
- **嵌套工具调用**：一个工具的输出作为另一个工具的输入参数
- **工具选择策略**：模型需判断 ① 是否需要调工具 ② 调哪个工具 ③ 参数怎么填

### Tool Description 设计最佳实践 ⭐

- **名称**：简洁、动作导向（search_web 而非 web_function）
- **描述**：说明工具的能力、适用场景、返回格式
- **参数**：每个参数带类型、说明、示例、约束（enum/range）
- **示例**：在描述中加入使用示例，显著提升调用准确率
- *核心问题*：工具太多（> 20个）模型会混淆怎么办？（分类分层 / 动态工具选择 / 用 RAG 检索相关工具）

### MCP 工具集成

- *核心思想*：通过 MCP 协议将工具定义从应用代码中解耦，实现工具复用
- *详见*：[MCP 协议](08_mcp.md)

## 记忆系统 🔥🔥 ⭐

### 短期记忆（对话上下文）

- **Sliding Window**：保留最近 N 轮对话
  - *问题*：N 太小丢信息，N 太大费 token
- **摘要压缩**：用 LLM 压缩历史对话，节省 token
  - *实现*：定期摘要 / 超长自动触发 / 层次化摘要
- **Token 预算管理**：根据任务复杂度动态调整上下文保留量
- *核心问题*：上下文窗口满了怎么办？（摘要压缩 vs 截断 vs 向量化存储）

### 长期记忆（外部存储）

- **向量数据库存储**
  - 存储历史对话/知识的 Embedding，按需检索
  - *适用*：语义相关的历史信息检索
- **知识图谱存储**
  - 结构化存储实体关系（用户偏好、事实知识）
  - *适用*：需要精确查询的结构化信息
- **结构化存储**（SQL/KV）
  - 用户偏好、任务状态、配置参数
  - *适用*：精确匹配、频繁读写的数据

### 工作记忆（Scratchpad）

- *核心思想*：Agent 的"草稿纸"，存储中间推理结果
- *实现方式*：
  - CoT 输出作为工作记忆
  - 专用 scratchpad token
  - 外部文件/变量（Cursor 的 notepad）
- *核心问题*：工作记忆和短期记忆的区别？（工作记忆是当前任务的中间状态，短期记忆是历史对话）

### 记忆更新策略 🔥

- **显式记忆**：用户明确要求记住的信息
- **隐式记忆**：Agent 自动从对话中提取值得记住的信息（偏好、纠正、重要事实）
- **遗忘机制**：过时/矛盾的记忆需要清理，避免干扰
- **记忆冲突解决**：新信息和旧记忆矛盾时，以新信息为准
- *核心问题*：如何设计记忆的"重要性评分"？（频率 × 时效 × 用户反馈）

## Agent 可靠性工程 🔥🔥

### 错误处理与重试

- **指数退避（Exponential Backoff）**：失败后等待时间指数增长
- **降级策略（Fallback）**：复杂方法失败，降级为简单方法
- **错误分类**：
  - 可恢复错误（API 超时、网络抖动）→ 重试
  - 不可恢复错误（参数错误、权限不足）→ 换策略 / 求助人类
  - 业务异常（搜索无结果、数据不存在）→ 调整查询

### Human-in-the-Loop ⭐

- *核心思想*：在关键决策点暂停，等待人工确认后继续
- *必须加的场景*：不可逆操作（删除/支付/发送邮件）/ 高风险决策 / 不确定性高时
- *实现模式*：
  - **审批模式**：每个关键操作都需人工确认
  - **信任梯度**：低风险自动执行，高风险需审批
  - **异常上报**：正常情况自动执行，异常情况才求助人类

### 成本控制 ⭐

- **Token 预算**：设置最大 token 数，超出时强制结束或摘要压缩
- **工具调用限制**：最大调用次数，防止无限循环
- **模型路由**：简单任务用小模型，复杂任务用大模型
- **缓存策略**：相同/相似查询复用历史结果
- *核心问题*：如何检测 Agent 陷入无限循环？（重复动作检测 + 进展评估 + 超时机制）

### 可观测性（Observability）🔥

- **Trace 追踪**：记录每一步的 Input/Output/Latency/Cost
- **Span 粒度**：LLM 调用 / 工具调用 / 检索 / 生成 各为独立 Span
- **日志结构化**：JSON 格式，包含 trace_id / step / action / result
- **工具支持**：LangSmith / Phoenix / Langfuse / Helicone
- *核心问题*：如何从 Trace 中快速定位 Agent 失败原因？（关注 LLM 输出的 Thought 和工具返回的 error）

## 多 Agent 协作 🔥🔥

### 协作模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **Leader-Worker** | 主 Agent 分解任务，子 Agent 并行执行 | 可并行的大任务 |
| **Debate（辩论）** | 多个 Agent 对同一问题给不同观点，最终综合 | 需多角度验证的决策 |
| **Peer Review** | 一个 Agent 生成，另一个 Agent 审查 | 代码/文本质量保障 |
| **Pipeline（流水线）** | Agent A 输出 → Agent B 输入 → Agent C 输入 | 有明确处理流程的任务 |
| **Blackboard（黑板模式）** | 所有 Agent 共享一个状态空间，各自贡献 | 复杂推理/知识整合 |

- *核心问题*：多 Agent 什么时候比单 Agent 好？（任务可并行 / 需要多角度验证 / 单个 Agent 上下文装不下）

### 通信机制

- **直接消息传递**：Agent A 直接发消息给 Agent B
- **共享状态**：所有 Agent 读写同一个状态对象（LangGraph 的 State）
- **事件驱动**：Agent 发布事件，其他 Agent 订阅响应
- *核心问题*：共享状态和消息传递哪种更好？（共享状态简单直观，消息传递更松耦合）

### 主流框架对比

| 框架 | 核心抽象 | 适用场景 | 特点 |
|------|----------|----------|------|
| **LangGraph** 🔥 | 图状态机（节点=Agent/工具，边=条件路由） | 复杂工作流、需要条件分支 | 灵活强大，学习曲线陡 |
| **CrewAI** | 角色扮演（Role/Goal/Backstory） | 团队协作场景、快速原型 | 简单易用，灵活性有限 |
| **AutoGen / AG2** | 对话式多 Agent | 需要 Agent 间自由讨论 | 灵活，但难以控制 |
| **MetaGPT** | 模拟软件公司（PM/架构师/程序员） | 软件开发流程 | 结构化强，场景限定 |
| **Swarm (OpenAI)** 🔥 | 轻量 Agent 切换（Handoff） | 简单的 Agent 路由 | 极简，仅做路由 |
| **Mastra** 🔥 | TypeScript Agent 框架 | Web 开发者友好 | 前端生态，新兴 |
