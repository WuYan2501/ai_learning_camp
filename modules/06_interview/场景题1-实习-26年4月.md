# 场景：从"几十个文档"到"几万文档"你的检索该如何做？

考核核心内容：方式是"系统设计 + 工程落地"，主要要考虑量级变化如何应对？非常考验是否有真实落地经验和落地实践的思考

## 结论先行，围绕结论进行展开

答：先简单说一下目前少量文档运行状况，然后带出思考如果增量会怎么处理。如下
原始简单架构由三层组成：数据层，检索层，生成层
量变后需考虑五层架构：数据层、索引层、召回层、重排层、生成层，主要是对检索层拆成了三层索引层 + 召回层 + 重排层，其实每一层都有相关升级改造，详细如下图

## 第一层：数据层
大家数据结构和来源都不同，实现细节都不同，但都逃这个问题-如何更新
1.如何保证大量数据更新？
答：少量数据比较简单，全量更新即可，数据量起来后全量不现实，时间长、embedding费用高等问题就无法接受。
所以需从增量考虑，但是如何检测源数据是否更新呢？
可以通过**基于 Hash 指纹的变更检测**，核心思路是给每个文档和每个 chunk 算一个 hash 指纹，然后自动化每日对比 hash 指纹

数据存储和抓取就得区分具体业务：第一种通过api获取，这种比较简单定时拉取计算即可；
第二种则是通过数据抓取例如原始 HTML/PDF，这种就需要加一个数据清洗层，将数据统一结构，例如json,在对结构话数据进行计算对比。

## 第二层：索引层
这一层其实真实业务场景倒不是考虑重点，一般都是使用专门的索引层服务进行存储，所以量起来了也是服务侧该解决的，但我们需要知道会有什么问题和解决思路
1.量变需要从单机到分布式部署
分布式则需考虑分区的实现，也就是分区策略，我举个例子：
数据可以按照文档类型分区（Partition）：法律和数学文档；怎么切分还得根据业务分析，有的类别拆的太大，就还得继续分级查分

区分完成后，路由层也是第三层的召回层，就是你需要设计的，判断这个 query 属于哪些 Partition，可根据语义、正则、精调模型等进行路由。

## 第三层：召回层
在第二层提到了路由层简单实现，现在对它进行丰富，让理解更加深刻和完整

前置路由尽可能轻量，常见用规则和关键词可以覆盖大部分场景，最好在搭配一个小模型分类器解决一些长尾场景，如果还是无法确定则通过LLM进行兜底判断。
路由输出按照下面结构，我拿学校教务系统举例子实现，
```
{
  "intent": "教师信息查询",
  "entities": {"学院": "计科", "姓名": "张三"},
  "target_partitions": ["teacher_info_计科"]
 }
```
这个返回包含了基本要素，意图、实体、分区信息

## 第四层：重排层
通过召回后，返回的数据中其实很有大量不相关知识被召回，为了减少干扰，需通过模型在精筛一遍。
也就是在利用 Cross-Encoder 对 (query, chunk) 逐对打分，选最高分前5～10条。
Cross-Encoder精排模型可以选用BGE-Reranker-v2-m3 或者 bge-reranker-large。
实际落地时，100个召回内容加上原始query在A10上大概 100-200ms 其实速度还是可以接受的。

这里面可以再设计一个缓存，高频重复查询就很适用

## 第五层：生成层
第一种是对简单问题快速答、困难问题慢思考。
根据用户query，来决定用什么模型
第二种需要对生成内容进行Fallback,例如无检索信息，则主动回复"暂无相关信息，请咨询相关人员"，不对回答编造
